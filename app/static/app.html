Instructions

Golden path — the usual flow

  1. Import — drag a CSV/XLSX onto the Import tab, or paste a shared-drive URL. Validation runs automatically right after.
  2. Check the results table — rows land in Valid, Invalid, Duplicate, Existing Customer, or Already Contacted. Nothing is dropped, everything is visible.
  3. Create or pick a campaign and queue the Valid rows into it — this also auto-generates a draft email per prospect. Campaign names are auto-suggested with today's date (e.g. "2026-07-20 — New Prospects") and must be unique — creating one with a name that already exists is rejected, so campaigns from different days never get confused with each other.
  4. Switch to the Campaigns tab to review queued drafts: edit subject/body, then Approve or Reject each one.
  5. Send the campaign — this delivers every Approved draft via Gmail. Requires email to be configured first (see Admin below).
  6. Watch progress on the Dashboard — the outreach funnel, replies, and audit log update live.
  7. When a prospect replies with interest, mark them Quote Requested, then Won or Lost from the Campaigns tab to track real business value.

Important details

  • Sending email requires setup. Go to the Admin tab → "Gmail App Password setup" and enter a Gmail address plus a Gmail App Password (not your real password — generate one at myaccount.google.com/apppasswords, requires 2-Step Verification). Takes effect immediately, no restart needed. (GMAIL_ADDRESS/GMAIL_APP_PASSWORD environment variables still work too, as a fallback.) Until configured, everything except the actual send/reply-poll works normally.
  • Suppression is checked twice — once when queuing into a campaign, again right before sending — so an opt-out can never slip through even if it happens after a draft was already approved.
  • Real sends are capped at a daily limit (Admin tab → "Daily send limit," default 20) — campaign sends and reply-draft sends combined, since Gmail's spam detection doesn't care which flow inside this app an email came from. "Send all approved" stops once the cap is hit for the day and leaves the rest Approved for the next run; simulated sends never count against it.
  • Opt-outs are also detected automatically from reply text (e.g. "unsubscribe," "remove me," "do not contact") and added to suppression without any manual step.
  • How a reply gets matched back to a sent email: by sender address only, not by message threading. Every poll cycle, the inbox is checked for unread mail from anyone currently in Sent status for some campaign; a match flips every Sent row for that address to Replied. It does not verify the message is actually a reply to that specific outreach email (no Message-ID/In-Reply-To check) — any new unread mail from that address while awaiting a reply counts.
  • Testing the funnel without Gmail connected: real sending and real reply-detection both hard-require Gmail credentials, by design — nothing ever pretends to send. To test the downstream funnel (Replied → Quote Requested → Won/Lost) and the dashboard math anyway, the Campaigns tab has "Simulate send (test)" on Approved drafts and "Simulate reply (test)" / "Simulate opt-out (test)" on Sent ones — same state transitions and audit trail as the real thing, clearly labeled and logged as simulated so they're never confused with a real send/reply. For a full-fidelity test of the actual Gmail send + IMAP matching code (not just the funnel math), the highest-confidence option is two real Gmail accounts: configure one as the sender, import the other as a test prospect, send for real, reply for real, then check /email/poll-now.
  • Smart replies to prospect questions: whenever a prospect replies (a real detected Gmail reply, or "Simulate reply (test)" with a reply body typed in), the tool automatically drafts a response — matched against the Q&A entries and stock categories in the Knowledge Base tab — and queues it on the Campaigns tab under "Reply drafts awaiting review." It's rule-based keyword matching, not an AI model, so every draft needs a quick human review before it sends; a draft with no strong match is clearly flagged "fallback" rather than guessing confidently. Approving sends it via Gmail immediately (fails clearly if Gmail isn't configured, same as any other send). Add more Q&A entries on the Knowledge Base tab to improve match quality over time.
  • Existing customers are matched against a seeded customer list and marked instead of queued for outreach.
  • Already-contacted prospects are caught on re-import. Email is the key: if an address was already sent an outreach email in any prior campaign — even from a completely different import file — re-importing it now flags it Already Contacted instead of validating as a fresh lead. Only prospects that have actually received a sent email are caught this way; someone still sitting Queued/Approved (never sent) in another campaign won't be flagged by this check.
  • Duplicates are flagged within the same import batch (same email seen twice in one file).
  • Data persists in a SQLite file across restarts as long as the server always runs with the same APEX_DB_PATH — back it up periodically since there's no separate hosted database.
  • The top-bar badge shows connection state: WAKING SERVER… means it's retrying while a sleeping backend (e.g. Render free tier after inactivity) spins back up — this can take up to ~90 seconds on the first load. DEMO MODE — SERVER UNREACHABLE means every retry failed — everything you do is kept only in this browser tab and is lost on refresh until the server responds; it keeps retrying quietly in the background and switches over automatically once reachable. CONNECTED TO SERVER means everything you do is saved for real.
